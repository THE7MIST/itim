Here are the types of Docker volumes explained clearly and practically:


---

🧠 Core idea

> Docker volumes are used to store data outside containers so it is not lost when containers stop or are deleted




---

🔹 1. Named Volume

What it is

Managed by Docker

Stored in Docker’s internal storage


Example

docker volume create mydata
docker run -v mydata:/data nginx

Use case

Persistent application data

Databases like MySQL, MongoDB


Key point

> Safest and most commonly used volume type




---

🔹 2. Anonymous Volume

What it is

Created automatically by Docker

No specific name given


Example

docker run -v /data nginx

Behavior

Docker creates random volume name

Hard to track/manage later


Use case

Temporary storage

Quick testing



---

🔹 3. Bind Mount

What it is

Maps a host directory directly to container


Example

docker run -v /home/user/app:/app nginx

Behavior

Container reads/writes directly to host folder


Use case

Development

Live code editing


Key point

> Most flexible but less isolated




---

🔹 4. tmpfs Mount

What it is

Stored in memory (RAM)

Not written to disk


Example

docker run --tmpfs /data nginx

Use case

Sensitive data

Temporary fast storage


Key point

> Data is lost when container stops




---

⚡ Comparison Table

Type	Stored Where	Persistence	Managed By	Use Case

Named Volume	Docker storage	Yes	Docker	Production data
Anonymous Volume	Docker storage	Yes (hard to track)	Docker	Temporary use
Bind Mount	Host filesystem	Yes	User	Development
tmpfs	RAM	No	Docker	Temporary secure data



---

🧠 Final understanding

Named → best for real apps
Anonymous → quick, temporary
Bind → direct host access
tmpfs → memory only


---

🎯 One-line answer

> Docker supports named volumes, anonymous volumes, bind mounts, and tmpfs mounts, each providing different ways to persist or manage data inside containers.

Lab: MySQL Data Persistence with and without Docker Volumes

Objective

Create a MySQL container

Insert data

Stop and remove container

Recreate container and observe data loss

Repeat using different volume types and verify persistence



---

Part 1: Without Volume (Data Loss Case)

Step 1: Run MySQL container

docker run -d \
--name mysql-test \
-e MYSQL_ROOT_PASSWORD=root \
-e MYSQL_DATABASE=school \
mysql:8


---

Step 2: Access MySQL

docker exec -it mysql-test mysql -uroot -proot

Inside MySQL:

USE school;
CREATE TABLE students (name VARCHAR(50), age INT);
INSERT INTO students VALUES ('Alice', 20);
SELECT * FROM students;


---

Step 3: Stop and remove container

docker stop mysql-test
docker rm mysql-test


---

Step 4: Run container again (same command)

docker run -d \
--name mysql-test \
-e MYSQL_ROOT_PASSWORD=root \
-e MYSQL_DATABASE=school \
mysql:8


---

Step 5: Check data

docker exec -it mysql-test mysql -uroot -proot

Inside:

USE school;
SELECT * FROM students;

Expected Result

ERROR: table does not exist

👉 Data is lost because no volume was used


---

Part 2: Using Named Volume (Data Persisted)

Step 1: Create volume

docker volume create mysqldata


---

Step 2: Run container with volume

docker run -d \
--name mysql-vol \
-v mysqldata:/var/lib/mysql \
-e MYSQL_ROOT_PASSWORD=root \
-e MYSQL_DATABASE=school \
mysql:8


---

Step 3: Insert data

docker exec -it mysql-vol mysql -uroot -proot

USE school;
CREATE TABLE students (name VARCHAR(50), age INT);
INSERT INTO students VALUES ('Bob', 25);
SELECT * FROM students;


---

Step 4: Stop and remove container

docker stop mysql-vol
docker rm mysql-vol


---

Step 5: Re-run container with same volume

docker run -d \
--name mysql-vol \
-v mysqldata:/var/lib/mysql \
-e MYSQL_ROOT_PASSWORD=root \
-e MYSQL_DATABASE=school \
mysql:8


---

Step 6: Verify data

docker exec -it mysql-vol mysql -uroot -proot

USE school;
SELECT * FROM students;

Expected Result

Bob  25

👉 Data persists because volume stores it outside container


---

Part 3: Using Bind Mount (Host Directory)

Step 1: Create folder on host

mkdir ~/mysql-data


---

Step 2: Run container with bind mount

docker run -d \
--name mysql-bind \
-v ~/mysql-data:/var/lib/mysql \
-e MYSQL_ROOT_PASSWORD=root \
-e MYSQL_DATABASE=school \
mysql:8


---

Step 3: Insert data

docker exec -it mysql-bind mysql -uroot -proot

USE school;
CREATE TABLE students (name VARCHAR(50), age INT);
INSERT INTO students VALUES ('Charlie', 30);


---

Step 4: Remove container

docker stop mysql-bind
docker rm mysql-bind


---

Step 5: Run again with same bind mount

docker run -d \
--name mysql-bind \
-v ~/mysql-data:/var/lib/mysql \
-e MYSQL_ROOT_PASSWORD=root \
-e MYSQL_DATABASE=school \
mysql:8


---

Step 6: Verify data

docker exec -it mysql-bind mysql -uroot -proot

USE school;
SELECT * FROM students;

Expected Result

Charlie  30


---

Part 4: Using Anonymous Volume

Step 1: Run container

docker run -d \
--name mysql-anon \
-v /var/lib/mysql \
-e MYSQL_ROOT_PASSWORD=root \
-e MYSQL_DATABASE=school \
mysql:8


---

Step 2: Insert data and remove container

(same steps as above)


---

Step 3: Re-run container

docker run -d \
--name mysql-anon \
-v /var/lib/mysql \
-e MYSQL_ROOT_PASSWORD=root \
-e MYSQL_DATABASE=school \
mysql:8

Expected Result

Data lost


👉 Because new anonymous volume is created


---

Final Observations

No volume → data lost
Named volume → data persists
Bind mount → data persists (host visible)
Anonymous → not reliable


---

Key Learning

Containers are ephemeral

Volumes store data outside container lifecycle

Named volume = best practice

Bind mount = development

Anonymous = temporary use

---

1. How to check volumes created

List all volumes

docker volume ls

What you will see

DRIVER    VOLUME NAME
local     mysqldata

👉 This shows only Docker-managed volumes

named volumes

anonymous volumes



---

Inspect a specific volume

docker volume inspect mysqldata

Example output (important part)

[
  {
    "Name": "mysqldata",
    "Driver": "local",
    "Mountpoint": "/var/lib/docker/volumes/mysqldata/_data"
  }
]

👉 Key field:

Mountpoint = where data is actually stored on your machine



---

2. Are all types of volumes listed?

docker volume ls shows:

Type	Visible here?

Named volume	Yes
Anonymous volume	Yes
Bind mount	No
tmpfs	No



---

Why?

Named + anonymous → managed by Docker → listed

Bind mount → just host folder → Docker does not manage it

tmpfs → memory only → no disk storage



---

3. Where named volume stores data

From inspect:

/var/lib/docker/volumes/mysqldata/_data


---

On Linux

cd /var/lib/docker/volumes/mysqldata/_data
ls

👉 You will see MySQL data files


---

On Windows / Mac

Docker runs inside a VM, so path is not directly visible

But you can still inspect:

docker volume inspect mysqldata


---

4. How to verify data is really stored there

Step 1: Create file inside container

docker exec -it mysql-vol bash

Inside:

cd /var/lib/mysql
touch testfile.txt
exit


---

Step 2: Check on host

ls /var/lib/docker/volumes/mysqldata/_data

👉 You will see:

testfile.txt


---

5. How to check which container is using a volume

docker ps

Then:

docker inspect mysql-vol | grep -A 5 Mounts


---

6. Summary (very important)

docker volume ls
→ shows only Docker-managed volumes

docker volume inspect
→ shows exact storage path

Named volume
→ stored in /var/lib/docker/volumes/

Bind mount
→ stored wherever you define

tmpfs
→ stored in memory


---

7. One-line clarity

> Named volumes are stored inside Docker’s internal directory and persist even after containers are deleted.
